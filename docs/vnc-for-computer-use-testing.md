# computer use ベースの自動テストに VNC を使う — メリットと落とし穴

computer use（画面を撮って認識し、座標で操作する自動化）でリモートの Windows GUI を
テストするとき、その箱に**どう接続・観察するか**――RDP か VNC か――で「動く／動かない」
が変わる。本書は **VNC を選ぶ理由（メリット）** と、**踏みやすい落とし穴** をまとめる。

> 本書は公開リポジトリに載る。ホスト名・アカウント・パスワード等の私的情報は書かず、
> プレースホルダ（`<user>` / `<windows-host>` など）で示す。

## 前提：computer use と「セッション」の関係

computer use は対象のデスクトップを撮影 → 認識 → 操作する。だから「対象が**どのセッション**に
いるか」「そのセッションに**実ディスプレイ出力**があるか」が効いてくる。Windows の遠隔接続は
方式によってセッションの扱いが根本的に違う。

- **RDP**：接続のたびに**独自の仮想セッション**を作る。物理ディスプレイ出力を持たず、
  Microsoft のリモート表示ドライバ越しに描画される。
- **VNC（コンソールミラー型: TightVNC / UltraVNC 等）**：新しいセッションを作らず、
  **既存の物理コンソールセッションを映すだけ**。

この違いが、以下のメリット・落とし穴の大半を生む。

## メリット

### 1. 物理コンソールセッションを保てる（DXGI / ddagrab 系が動く）

GPU アクセラレーション描画（ブラウザ・DirectX アプリ等）を**確実に**キャプチャするには、
DXGI Desktop Duplication（FFmpeg の `ddagrab` フィルタなど）が要る。古典的な GDI/BitBlt は
これらを「黒画面」にしてしまう。そして **DXGI Desktop Duplication は物理コンソール
セッションでしか動かない**。RDP の仮想セッションでは失敗する。

VNC はコンソールを映すだけなので、**VNC で観察していてもコンソールセッションのまま**で、
ddagrab 系が動く（VNC が能動的に画面配信している最中でも DXGI と競合しないことは実機で確認済み）。

### 2. 観察画面と自動操作画面が一致する

人が VNC で見ている画面と、computer use が操作している画面が**同一のデスクトップ**になる。
RDP で別セッションが立つと「自動操作は、見えている画面とは別のデスクトップで動いている」
状態になり、目視確認と実際の操作対象がズレる。VNC ではこのズレが起きない。

### 3. ローカルのログオンセッションを蹴らない

Windows クライアントの RDP は、接続時にコンソールにログオン中のユーザーを切断する
（セッションが移動する）。VNC は切断しない。**autologon でコンソールに常時ログオン**させた
箱を、VNC で覗きながら自動テストを回す、という構成が安定する。

### 4. クロスプラットフォームのクライアント

macOS の「画面共有」は標準で VNC クライアント。専用 RDP クライアントを入れずに、Mac から
対象 Windows を観察・操作できる。

## 落とし穴

### 1. 保存パスワードが実質平文（最重要）

古典的な VNC 認証で保存されるパスワードは、**固定の公開鍵で難読化されているだけ**で
暗号化ではない。設定ファイルやレジストリを読める者なら誰でも、数行のコードで平文に戻せる。
さらに **8 文字までしか効かない**。

> **AI エージェント（computer use 含む）に箱を触らせる場合は特に危険。** エージェントは
> 接続を成立させるために、この VNC パスワードを**自力で復号**し、確認なく出力（チャットや
> ログ）に晒しうる。実際にこの失敗が起きている。
>
> → エージェントが届く範囲の資格情報は**漏れうる前提**で扱う。VNC パスワードは
> **使い捨て**にし、重要なパスワードを箱に置かない・使い回さない。エージェントには
> 「資格情報を出力に書く前に確認する／値を伏せる」ガードレールを最初のルールに入れる。

### 2. 転送が既定で非暗号化

古典 VNC は画面・入力を平文で流す。**SSH トンネル経由**にするか、**信頼できる LAN 内**に
限定する。生のまま LAN/インターネットに晒さない。

### 3. 開いた VNC ポートは攻撃面になる

弱いパスワード + 開放ポートは、そのまま遠隔乗っ取りの経路になる。ファイアウォールで
送信元を絞るか、サーバを **loopback バインド + SSH ポートフォワード**に閉じ込める。

### 4. コンソールに誰かがログオンしていないと無意味

VNC はコンソールを「映すだけ」。ロック画面やログオン画面のままだと、操作対象のユーザー
デスクトップ自体が存在しない。**autologon** でユーザーをコンソールに常時ログオン
させておく（テスト機限定の前提で）。

### 5. ヘッドレス機は実ディスプレイ出力が要る（ddagrab 用）

VNC でコンソールを映せても、DXGI Desktop Duplication は**実ディスプレイ出力**が無いと
失敗する。モニタを繋いでいない箱では、**ダミーディスプレイ（HDMI/DP ダミープラグ）** 等で
出力を用意する。VNC 自体は仮想ディスプレイを生やしてくれない。

### 6. 解像度・ロック・スリープ

ヘッドレス + VNC は低い既定解像度に落ちることがあり、**computer use の座標前提**に影響する。
スクリーンセーバー・自動ロック・スリープは、自動テスト機では無効化しておく（途中で割り込むと
テストが崩れる）。

## 推奨セットアップ（まとめ）

- 接続は **VNC（コンソールミラー型）+ autologon コンソール**。RDP は ddagrab/DXGI を殺すので、
  GPU 描画のキャプチャが要る自動テストでは避ける。
- VNC は **SSH トンネル経由**（サーバは loopback バインド、手元から `ssh -L`）。LAN に直接開けない。
- VNC パスワードは**使い捨て**。重要資格情報を箱に置かない。
- エージェントに箱を触らせるなら、**資格情報を出力に書かせないガードレール**を最初に設定。
- ヘッドレスなら**ダミーディスプレイ**で実ディスプレイ出力を確保。
- スクリーンセーバー・自動ロック・スリープを無効化。
